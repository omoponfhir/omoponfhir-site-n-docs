Need Some More Help?

Mappings: The largest barrier to leveraging FHIR and OMOP CDM together lies in the mapping between them. A one to one mapping is impossible, and there are many cases with a single FHIR Resource may map to multiple tables or columns within OMOP CDM, and sometimes vice versa. Mapping performed by OMOPonFHIR takes this into account, and provides a robust approach leveraging years of experience and user feedback.

Standard Deployment: As a Spring based application, OMOPonFHIR is built into a Java Web Application Resource (WAR) file. In order to build the WAR, users will need to select their desired versions of FHIR and the OMOP CDM and configure the Maven pom.xml configuration file to reflect the selected versions and the appropriate mapping layer in turn. For example, in FHIR R4 to OMOP CDM v6.0, the FHIR R4 server layer along with the OMOP CDM v.6.0 database layer must be specified, as well as the FHIR R4 to OMOP CDM v6.0 specific mapping layer. Once built, OMOPonFHIR can be deployed in any environment which supports WARs, predominantly through an Apache Tomcat server directly. An OMOP CDM database must be provided separately. For more information on configuration, please see the OMOPonFHIR repositories.

Docker Deployment: To support environments which allow Docker containers, a Dockerfile is included with OMOPonFHIR. This allows the deployment of OMOPonFHIR through Docker directly or anywhere in which Docker images may be run, such as through Kubernetes. The Maven build process is handled within the container, requiring only version based component configuration to be made in the Maven pom.xml and environment variables to be provided through the Dockerfile. This approach can be utilized with any OMOP CDM database, including one which is also containerized. It is recommended that when being ran directly with Docker that containers are setup with bridge networking. For more information on building and running an OMOPonFHIR image, please see the OMOPonFHIR repositories.
